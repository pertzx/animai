## CONTEXTO DO PROJETO

O AnimAI é um editor de vídeo com IA que roda 100% localmente no dispositivo do usuário. 
O servidor serve APENAS para autenticação, dados do usuário e chamadas de IA quando necessário. 
Toda a análise semântica de vídeo, imagem e áudio é processada localmente via navegador.

### Hardware alvo primário (meu PC de desenvolvimento)
- Notebook Samsung, 4 GB RAM, CPU dual-core, sem GPU dedicada, HDD lento
- Vídeo de 14 segundos demora 60–90 segundos para analisar
- O sistema trava frequentemente por OOM (Out of Memory) e swap no disco

### Escopo de plataforma (meta de distribuição)
1. **Desktop Web**
2. **Desktop App WebView**
3. **Android WebView**

**FORA DO ESCOPO:** iOS, Safari, WKWebView, Firefox mobile. Não precisa otimizar para esses.

## PONTOS DE ATENÇÃO ESPECÍFICOS (identificados em análise prévia)

### A. GARGALOS DE PERFORMANCE JÁ MAPEADOS

1. **Seek aleatório no `<video>` element** (`frame-source.ts`)
   - `video.currentTime = t` a cada frame força redecodificação de GOPs inteiros
   - Em CPU fraca + HDD, cada seek leva 200–500ms
   - 14 frames × seek = 2.8–7 segundos só em seek

2. **Cessão de thread a cada frame** (`orchestrator.ts`)
   - `await yieldToUi()` (setTimeout(..., 0)) a cada frame
   - Overhead de 15–30ms por yield em CPU fraca
   - Quebra pipelining de CPU/GPU

3. **Todos os modelos carregados simultaneamente** (`orchestrator.ts`)
   - `Promise.all(plugins.map(p => p.init()))` carrega 4+ modelos MediaPipe de uma vez
   - 4 modelos × ~300 MB = 1.2 GB+ residentes em RAM
   - Em 4 GB, isso força swap e OOM

4. **Decode de áudio bloqueante** (`run.ts`)
   - `await decodeBlobAudio(blob)` executa ANTES de iniciar a análise de vídeo
   - Áudio e vídeo não se sobrepõem

5. **Amostragem uniforme sem filtro de redundância** (`frame-source.ts`)
   - Frames em intervalos fixos, sem scene detection ou detecção de frames idênticos
   - Em entrevistas/podcasts, 60–80% dos frames são visualmente idênticos

6. **OCR em todos os frames amostrados** (`plugins/ocr.ts`)
   - Tesseract.js: 1–3s por frame em CPU fraca
   - Texto em vídeo persiste por segundos — não precisa de OCR a cada frame

7. **`createImageBitmap` por frame** (`frame-source.ts`)
   - Aloca textura GPU/VRAM a cada frame
   - 14 frames × ~400 KB = 5.6 MB de alocação/liberação por vídeo
   - Força GC constante em 4 GB RAM

8. **Finalização síncrona e sequencial** (`orchestrator.ts`)
   - `for (const p of plugins) p.finalize(raw, context)` roda na main thread
   - Tracking pode ser O(N²) em CPU fraca

9. **Cache apenas do JSON final** (`store.ts`)
   - Se usuário muda 1 plugin, todo o vídeo é reprocessado
   - Não há cache granular por plugin

10. **Ausência de detecção de hardware** (`config.ts`)
    - Modo "low" ainda pode ser agressivo demais para 4 GB RAM
    - Não há modo "critical" ou "mobile"

### B. PROBLEMAS DE ARQUITETURA EXTRAS (não relacionados ao Semantic Analyzer diretamente, mas que afetam performance geral)

11. **Blob inteiro do vídeo guardado em memória** (`project-store.ts` / `media-storage.ts`)
    - O sistema guarda o `Blob` completo do vídeo na memória do navegador
    - Um vídeo de 100 MB fica residente em RAM o tempo todo
    - Deveria usar FileSystemAccess API para guardar path/reference, não o blob
    - Isso compete diretamente com a RAM usada pelo Semantic Analyzer

### C. FEATURES FUTURAS QUE PRECISAM SER CONSIDERADAS NA ARQUITETURA

12. **Sistema de câmera 3D real** (como Alight Motion)
    - Câmera virtual com profundidade, parallax, perspectiva
    - Elementos em camadas com Z-depth
    - Não é simples zoom/pan — é transformação 3D real de câmera
    - Deve ser considerado no canvas de preview e no sistema de keyframes

13. **Keyframes universais** — qualquer propriedade de qualquer elemento
    - Hoje keyframes parecem limitados a propriedades específicas
    - Meta: qualquer propriedade numérica, booleana ou de transformação de qualquer elemento (vídeo, texto, shape, imagem, efeito) deve ser animável via keyframes
    - Isso inclui: posição X/Y/Z, escala, rotação, opacidade, cor, blur, brilho, propriedades de câmera (FOV, depth of field), etc.

## RESTRIÇÕES

- NÃO quebrar a API pública dos plugins (init / analyzeFrame / analyzeSequence / finalize / dispose)
- NÃO alterar o formato do JSON final (`SemanticTimeline`) — salvo exceções justificadas
- NÃO remover suporte a nenhum plugin existente — apenas otimizar como/when rodam
- Manter compatibilidade com AbortSignal para cancelamento
- Manter compatibilidade com callbacks de progresso (onProgress, onFrame, onEvents, onDetectorStatus)
- Preservar a capacidade de "modo econômico / balanceado / avançado / manual"

## INFORMAÇÃO ADICIONAL

- O projeto usa Bun como package manager
- Monorepo com apps/web, apps/server, apps/image
- O Semantic Analyzer está em `apps/web/src/services/semantic/`
- MediaPipe Tasks Vision é usado para visão (face, pose, hands, object, scene)
- Tesseract.js é usado para OCR
- Whisper (via worker) é usado para STT/speech
- Web Audio API é usada para análise de áudio/música
- IndexedDB (via Zustand + idb) é usada para cache de timelines
